# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-03-12
**Phases:** 1–4
**Total Plans:** 9

## Overview

A Telegram bot that proactively monitors your self-hosted ActualBudget instance, detects uncategorized transactions, and asks you to categorize them via inline keyboard buttons. Runs on your Coolify server alongside Actual.

Delivered in 4 phases over 2 days (2026-03-11 → 2026-03-12), moving from security foundation to full automation.

## Phases

### Phase 1: Authorization

**Goal:** Only authorized Telegram users can interact with the bot.
**Depends on:** None (foundation phase)
**Plans:** 2 plans

Plans:
- [x] 01-01-PLAN.md — Project setup, Node.js/TypeScript/grammY/Docker skeleton
- [x] 01-02-PLAN.md — Authorization middleware with TELEGRAM_USER_ID validation

**Details:**
- Authorization middleware validates user ID at grammY middleware layer
- Unauthorized users silently ignored with security logging
- Bot fails fast if TELEGRAM_USER_ID not configured
- `/start` command shows user's Telegram ID for easy setup
- Docker container ready for Coolify deployment

---

### Phase 2: Integration

**Goal:** Bot connects to ActualBudget and fetches uncategorized transactions and categories.
**Depends on:** Phase 1
**Plans:** 2 plans

Plans:
- [x] 02-01-PLAN.md — Install @actual-app/api, create ActualApiService wrapper
- [x] 02-02-PLAN.md — Integrate API into bot with credential validation

**Details:**
- ActualApiService class encapsulates all ActualBudget API calls
- Connects via server URL + password, downloads budget by BUDGET_ID
- Fetches uncategorized transactions from all on-budget accounts
- Fetches categories and category groups
- Payee cache and account cache for efficient lookups
- E2E encryption password support (optional)

---

### Phase 3: User Interface

**Goal:** Users can categorize transactions via inline keyboard buttons with 2-tap workflow.
**Depends on:** Phase 2
**Plans:** 3 plans

Plans:
- [x] 03-01-PLAN.md — Enhance API with updateTransaction
- [x] 03-02-PLAN.md — Transaction display command with category buttons
- [x] 03-03-PLAN.md — Category button callback handler

**Details:**
- `/transaction` command displays first uncategorized transaction
- Inline keyboard groups categories by group (visual hierarchy with 📁 headers)
- 2-column category layout, 2 buttons per row
- SessionManager stores transaction data (avoids Telegram 64-byte callback limit)
- Callback format: `cat_{sessionId}_{categoryId}`
- Success message confirms payee → category with amount
- `/categories` command shows full grouped category list
- `/uncategorized` shows count of uncategorized transactions
- `/help` shows all available commands

---

### Phase 4: Automation

**Goal:** Bot automatically checks for uncategorized transactions every 4 hours and notifies user.
**Depends on:** Phase 3
**Plans:** 2 plans

Plans:
- [x] 04-01-PLAN.md — NotifierState service and PollScheduler
- [x] 04-02-PLAN.md — Notification formatting and verification

**Details:**
- NotifierState persists notified transaction IDs to JSON file (`/tmp/actual-telegram-notif/notifier-state.json`)
- PollScheduler runs immediate check on startup + 4-hour interval (14,400,000 ms)
- Skip-if-busy mutex prevents concurrent poll cycles
- Bank sync via `runBankSync()` before each poll (with error tolerance)
- Sorts by date descending, notifies on newest unnotified transaction
- Notification reuses exact same format/keyboard as `/transaction` command
- State survives bot restarts

## Milestone Summary

**Key Decisions:**

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| grammY framework | TypeScript-native, modern API, excellent type safety | ✅ Good |
| @actual-app/api | Official ActualBudget client, same pattern as actual-ai | ✅ Good |
| Single authorized user (TELEGRAM_USER_ID) | Solo tool — multi-user adds complexity with no benefit | ✅ Good |
| SessionManager for callbacks | Telegram 64-byte limit on callback_data | ✅ Good |
| File-based state persistence | Simple, no DB dependency, works in Docker | ✅ Good |
| 4-hour fixed poll interval | Not spammy, catches transactions before next check | ✅ Good |
| Skip-if-busy mutex | Simple, appropriate for 4-hour interval | ✅ Good |
| Newest-first notification | Most relevant transaction surfaced first | ✅ Good |
| buildCategoryKeyboard() utility | Eliminates ~80-line duplication between command and poller | ✅ Good |

**Issues Resolved:**

- Payee names were displaying as UUIDs (resolved via payee cache + payee map lookup)
- Category groups not displayed (resolved via getCategoryGroups() with `any` cast)
- Account names not shown in transaction form (resolved via getAccount() with cache)
- Success message didn't show payee name after categorization (resolved via session data)
- `updateTransaction` was scanning all accounts to verify existence (removed — unnecessary)

**Technical Debt:**

- `stopPolling()` exported but never wired to SIGTERM handler (graceful shutdown not implemented)
- Payee cache never invalidated (acceptable for short-lived bot sessions)
- Date range in `getTransactions` is hardcoded to 1990-2030 (pragmatic, not configurable)
- `(actual as any)` casts for `getCategoryGroups` and `runBankSync` (API types incomplete)
